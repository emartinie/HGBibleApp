# Knowledge Module Prototype Schema

## Purpose

A knowledge module is a reusable relationship object.

It should be openable from many systems without rewriting content for each destination. A module is not a sermon, card, commentary, route, or UI. It is structured knowledge plus typed relationships.

This prototype uses one example:

- `data/modules/greater-exodus.json`

## Core Principle

The module owns the theme identity and relationship map.

Other systems own their own experiences:

- NT owns chapter reading.
- Sacred Time owns appointed-time data.
- Timeline owns chronological events.
- Intertext owns Scripture-to-Scripture relationships.
- Commandments owns commandment records.
- Maps owns geography.
- Sources owns source metadata.
- Orbit owns contextual launch actions.

## Top-Level Fields

`schemaVersion`

Version of this prototype shape.

`moduleType`

The kind of module. Current example: `knowledge-theme`.

`id`

Stable module id. Recommended form: `theme.slug`.

`slug`

URL/search-friendly identifier.

`title`

Human-readable name.

`status`

Draft/prototype/review/ready state.

`summary`

Short non-commentary description with:

- `what`
- `whyItMatters`
- `boundaries`

`primaryQuestions`

Questions the module helps answer. These guide discovery and UI prompts without becoming commentary.

`tags`

Search and grouping terms.

`confidenceScale`

Local definitions for relationship confidence labels.

`relationships`

The main payload. Relationships are grouped by destination system or relationship family.

`reviewQueue`

Known issues, disputed claims, missing dependencies, or owner decisions.

## Relationship Object Shape

Each relationship should be small and typed.

Common fields:

- `id`: stable relationship id.
- `title` or `label`: human-readable name.
- `reference`: Scripture reference when applicable.
- `relationshipType`: why this target relates to the module.
- `confidence`: confidence label.
- `note`: optional restraint or review note.

Special fields may appear by group:

- `from` and `to` for Intertext.
- `path` for local source files.
- `type` for source classification.

## Confidence Labels

Recommended labels:

- `explicit`: directly stated by the cited text.
- `strong`: strong relationship based on direct wording, repeated biblical pattern, or close canonical link.
- `medium`: reasonable thematic or typological relationship, but not directly stated.
- `disputed`: recognized interpretive disagreement or multiple competing readings.
- `speculative`: possible association requiring review before reuse in study surfaces.

## Relationship Groups

### `scriptures`

Direct Scripture references that define or strongly shape the module.

### `relatedThemes`

Other reusable modules this one should connect to.

### `sacredTime`

Appointed-time entries related to the module.

### `timelineEvents`

Chronological events related to the module.

### `revelationChapters`

Revelation chapter anchors related to the module.

This group is not Revelation-specific forever; a later schema may generalize this as `bookChapters`.

### `commandments`

Commandment records related to the module.

### `intertext`

Scripture-to-Scripture edges.

### `sources`

Sources that discuss, support, discover, or dispute the module.

### `maps`

Map layers or geography targets.

### `orbitActions`

Contextual actions Orbit may show later. Orbit launches these; it does not own the module.

## What This Prototype Proves

One module can answer:

- What is this theme?
- Why does it matter?
- Which Scriptures directly relate?
- Which related themes exist?
- Which Sacred Time entries relate?
- Which Timeline events relate?
- Which Revelation chapters relate?
- Which Commandments relate?
- Which Intertext entries relate?
- Which Sources discuss it?
- Which Maps relate?
- What confidence labels belong to each relationship?

## What This Prototype Does Not Do

- It does not modify existing cards.
- It does not change production behavior.
- It does not integrate with NT, Sacred Time, Timeline, Intertext, Search, or Orbit.
- It does not populate a full module library.
- It does not write commentary.
- It does not resolve disputed eschatology.

## Open Schema Questions

- Should `revelationChapters` become a generic `scriptureLocations` group?
- Should every relationship include `direction`, such as `supports`, `echoes`, `launches`, `dependsOn`, or `contrastsWith`?
- Should confidence labels be global app constants rather than module-local definitions?
- Should sources be normalized in a separate source registry?
- Should Intertext edges live only in Intertext data and be referenced here by id?
- Should Orbit actions be stored in modules or generated by the host system?
